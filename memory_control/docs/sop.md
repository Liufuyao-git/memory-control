# SOP — 系统配置文件瘦身

## 说明
- **执行者**：AI 助手
- **前提**：用户说出触发词后进入本 SOP，Phase 2 执行必须等用户说「执行」
- **自身成本**：Phase 0 读取所有文件，未精简状态下约消耗 **~5,000 tok**（保守上限）

---

## Phase 0：数据采集（自动）

**步骤顺序严格执行，不得跳步：**

1. 运行审计脚本：
   ```bash
   python3 scripts/audit.py
   ```
   （在仓库根目录执行；若配置文件在其他工作区根下，请先 `cd` 到该根目录或设置 `OPENCLAW_WORKSPACE`。）

2. **检查输出中的 `abort` 字段**：
   - `abort: true` → 立即输出告警原因，**中止 Phase 0，不进入 Phase 1**
   - `abort: false` → 继续执行

   常见中止原因：USER.md 不存在或内容过少。处理方式：提示用户补充 USER.md 后重新触发 skill。

3. **检查 `user_baseline_status`**，提炼用户场景基线（仅模型侧推理，不逐字输出给用户）：
   ```
   用户场景基线（示例结构）：
   - 使用渠道：网页端单人对话（无群聊 / 语音播报 / 第三方社区机器人）
   - 常用系统：关系型数据库、代码托管、工单或消息类 API（按 USER.md 实际填写）
   - 不使用：语音通道、表情互动协议、境外即时通讯等（按 USER.md 实际填写）
   ```
   后续每块内容审计时，对照此基线判断「是否与用户场景相关」。

4. 读取各配置文件全文（按 audit.py 输出的 `exists=true` 文件列表）

5. 读取 `docs/techniques.md`

## Phase 1：审计报告（输出给用户，等确认）

### 1. 总览表

| 文件 | 当前 tok | 目标 tok | 可节省 | 主要技巧 | 凭据保护 |
|------|---------|---------|--------|---------|---------|
| ... | ... | 保守估算 | ... | ... | ✅行N,M / — |

**目标 tok 估算说明**：删除所有识别到的冗余块后的剩余体积估算。
- 若文件带 `cjk_warning: true`，误差可达 ±25%，必须在报告中标注「实际节省以 Phase 2 执行后重测为准」
- 不含中文文件误差约 ±15%，同样标注「保守估算」

### 2. 逐文件分析

每个文件按 `techniques.md` 对应视角分析，每条建议必须：
- **引用原文**：附上被建议删除/压缩的原始文字片段（前 60 字符），让用户可以快速核对
- **说明冗余类型**：场景无关 / 跨文件重复 / 死代码 / 冗余表述 / 生命周期终止
- **凭据保护**：只保护 `credential_lines[]` 里的行（`likely_credential=true`）；`credential_lines_noise[]` 里的行是误报，正常处理，不保护

输出格式示例：
```
AGENTS.md（当前 1,540 tok → 目标 ~280 tok，可节省 ~1,260 tok）

| 块 | 原文片段 | 行数 | 冗余类型 | 建议 |
|----|---------|------|---------|------|
| 群聊行为规范 | "## 群聊（何时发言）\n当有人..." | 25 行 | 场景无关 | 删除 |
| 心跳 vs 定时任务 | "## 心跳 vs 定时任务\n心跳..." | 20 行 | 跨文件重复（HEARTBEAT.md 已有） | 改为一行引用 |
```

### 3. 优先级建议

按 audit.py 输出的 `severity` 排序（High → Medium → Low），同级按节省 token 数降序。

| 优先级 | 对应 severity | 建议时机 |
|--------|-------------|---------|
| P0 | High | 立即处理 |
| P1 | Medium | 本轮处理 |
| P2 | Low | 下次处理 |

### 4. 可选建议项（不属于文件审计，用户自行决定）

在审计报告末尾附上以下建议，**AI 不自动执行，由用户决定是否采纳**：

**建议 A：会话边界纪律**
> 对话超过 20 轮或明显切换主题时，建议主动开新会话（`/new`）。长会话历史每轮重新计费，是仅次于配置文件的第二大成本来源。如需写入 AGENTS.md 永久生效，请明确告知。

**建议 B：memoryres/ 冷存储层**
> 如果 MEMORY.md 有低频引用的大段内容（OKR / 历史记录 / 参考表格），可迁移到 `memoryres/` 按需加载，内容不丢失但不占常驻上下文。如需执行，请说「执行 memoryres 迁移」。

## Phase 2：执行（必须等用户明确说「执行」才触发）

### 第一步：调用备份脚本（强制，不可跳过，失败则中止）

```bash
bash scripts/backup.sh
```

- 看到输出第一行为 `BACKUP_OK <路径>` → 继续
- 看到 `BACKUP_FAILED <原因>` → **立即中止，输出「备份失败，Phase 2 中止，原因：<原因>」，不执行任何文件修改**

### 第二步：逐文件修改

按 Phase 1 确认的建议逐文件执行。

### 第三步：重新运行 audit.py，验证实际节省

```bash
python3 scripts/audit.py
```

对比 Phase 0 和 Phase 2 后的 token 数，输出实际节省量（不再依赖估算值）。

### 第四步：写变更记录

追加写入工作区 `memory/YYYY-MM-DD.md`（路径随 `OPENCLAW_WORKSPACE` 而定），格式：

```markdown
## 系统配置文件瘦身（HH:MM）

| 文件 | 优化前 tok | 优化后 tok | 节省 | 主要操作 |
|------|-----------|-----------|------|---------|
| AGENTS.md | 1,540 | 280 | -1,260 | 删群聊/TTS/表情块，改为引用 |

**合计节省：X tok/轮（实测） · 万轮节省 ≈ $Y**

### 凭据保护（跳过的行）
- TOOLS.md 第 N 行（GitLab PRIVATE-TOKEN）：只压缩注释，token 值不动

### 被删除的主要内容摘要
- AGENTS.md「群聊行为规范」（25 行）：用户仅单人对话，删除群聊规范
```
