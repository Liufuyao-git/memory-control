# 记忆管理库 — 系统配置文件

## 使用方式
AI 在 Phase 1 审计时，根据文件类型选择对应视角逐块分析。
每种视角有**核心问题**（逐块问自己）+ **操作方法** + **实战案例**。

---

## 视角一：逐块问诊（AGENTS.md 类 — 行为规范文件）

**核心问题**：这个块，用户会触发它的场景吗？另一个文件已经有这块内容了吗？

**操作方法**：
1. 逐块扫描文件，每块问两个问题
2. 「不会触发」→ 直接删
3. 「另一个文件已有」→ 改为一行引用，删正文

**实战案例（2026-04-10）**：
| 块 | 原因 | 处理 |
|----|------|------|
| 群聊行为规范（25 行） | 用户只用 webchat 单人对话 | 直接删 |
| 表情回应规则（15 行） | Discord 功能，用户不使用 | 直接删 |
| 语音/平台格式化（12 行） | TTS/WhatsApp，用户不使用 | 直接删 |
| 心跳 vs 定时任务（20 行） | HEARTBEAT.md 已有 | 改为一行引用 |
| 记忆维护（15 行） | SOUL.md 已有 | 改为一行引用 |
| BOOTSTRAP 节（5 行） | 触发条件已不存在 | 直接删（死代码） |
| 评审循环节 | 职责移到专属 SKILL.md | 一行引用 |

参考量级：大型 AGENTS.md 可压缩 60~85%

---

## 视角二：按需加载检查（TOOLS.md 类 — 工具配置文件）

**核心问题**：这条配置，是每轮对话必需的，还是只在特定 skill 调用时才需要？

**操作方法**：
1. 把文件内容分为两类：
   - **全局必需**（每轮都用）：系统路径、登录态位置、全局凭据 → 保留
   - **按需加载**（偶尔用）：特定 skill 的调用规范、API 行为说明 → 移到对应 SKILL.md
2. 凭据类（token/key/secret）：**只压缩表述，不移除位置**

**实战案例（2026-04-10）**：
- TOOLS.md 原有大量「内部 HTTP API / 消息网关」类调用说明 → 全部移到各自 SKILL.md
- 保留：SSO token 路径、代码托管凭据、workspace 路径、登录页 URL
- 参考量级：TOOLS.md 可压缩 50~75%（取决于内联了多少 skill 规范）

---

## 视角三：单一职责检查（MEMORY.md 类 — 记忆文件）

**核心问题**：这条内容，是「发生了什么」（事件/决策/状态），还是「应该怎么做」（行为规范）？

**操作方法**：
1. 扫描文件，识别混入的行为规范类内容
2. 行为规范 → 移到 SOUL.md 或 AGENTS.md，MEMORY.md 只留引用或直接删
3. **历史事件记录不动**（即使内容很长，那是记忆，不是冗余）

**实战案例（2026-04-10）**：
- MEMORY.md 原有「报告生成设计准则」等行为规范描述 → 移到 SOUL.md
- 保留：踩坑记录、项目状态、代码仓库约定
- 参考量级：MEMORY.md 可压缩 20~50%（行为规范占比越高，收益越大）

---

## 视角四：冗余表述压缩（USER.md / IDENTITY.md 类 — 档案文件）

**核心问题**：这条信息，能不能用更少的词表达同样的意思？

**操作方法**：
1. 找到多段解释同一件事的内容，合并为一行定义
2. 删除「为什么」的解释——AI 只需要「是什么」
3. 删除过时字段（已不使用的联系方式、已变更的职责描述）

**实战案例（2026-04-10）**：
- USER.md 原有角色描述三段解释 → 压缩为一行
- 参考量级：USER.md/IDENTITY.md 可压缩 30~60%

---

## 视角五：生命周期终止（BOOTSTRAP / 引导类文件）

**核心问题**：这个文件的存在目的是什么？这个目的是否已经完成？

**操作方法**：
1. 识别一次性文件（bootstrap、setup、onboarding、引导脚本）
2. 确认其使命已完成（初始化已做、身份已建立、配置已固化）
3. 整体删除，写删除原因到变更记录

**实战案例（2026-04-10）**：
- BOOTSTRAP.md：首次上线引导脚本，IDENTITY.md/SOUL.md/USER.md 全部建立后使命完成
- 整体删除，节省 326 tok/轮

---

## 视角六：冷存储迁移（memoryres/ 分层架构）

**核心问题**：这块内容，上周被引用过吗？不到一次/周的内容，值得常驻加载吗？

**什么是 memoryres/**

`~/.openclaw/workspace/memoryres/` 是「冷存储层」，介于「始终加载」和「几乎不用」之间：
- **始终加载**：MEMORY.md / SOUL.md 等 bootstrap 文件，每轮必须读入
- **按需加载（冷存储）**：memoryres/*.md，AI 判断当前任务需要时主动 `read`，平时不占上下文
- **原始日志**：memory/YYYY-MM-DD.md，极少加载

AI 在每次会话中，遇到相关任务时会主动说「让我查一下 memoryres/xxx.md」并读取，不需要用户干预。

**迁移判断标准（频率阈值）**：

| 引用频率 | 建议处理 |
|---------|---------|
| 几乎每轮都用 | 保留在原文件 |
| 约 1 次/周 | 迁移到 memoryres/ |
| 很少被提起 | 迁移到 memoryres/ 或直接删 |

**操作方法**：
1. 识别符合迁移条件的块（参考表格 + 代码引用 + 历史积累的大段说明）
2. 创建目标文件：`memoryres/[主题名].md`（如 okr.md / tech-debt.md / incidents.md）
3. 把内容移入新文件
4. 在原文件该位置替换为一行指针：`→ 详见 memoryres/[filename]`
5. **含凭据的块禁止迁移**（见下方凭据保护规则），只能原地压缩

**适合迁移到 memoryres/ 的内容类型**：
- OKR / 目标表格
- 词汇表 / 术语对照
- 技术债务 backlog
- 历史故障记录
- CSS / 代码参考片段
- 「待确认」类 pending 事项

**实战参考**：token-slim 实测，把低频内容迁入 memoryres/ 后，MEMORY.md 从 >200 行压缩到 <150 行，每轮节省约 800~1,500 tok，内容零丢失。

---

## 凭据保护规则（所有视角通用）

audit.py 标记 `has_credentials=true` 的文件，执行以下规则：

- 含 `token / secret / key / password / PRIVATE-TOKEN` 关键词的**行或块**：只压缩表述，不移除
- 可以：把多行凭据说明压缩为一行注释
- 不可以：把凭据路径或 token 值移到别处或删除
- **不可以：把含凭据的块迁移到 memoryres/**（迁移≠删除，内容仍会被加载，安全边界不变）
